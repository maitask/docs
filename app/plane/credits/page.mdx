# Credits And Payments

## Scope

Plane credits provide deterministic execution billing with auditable balance changes and provider-backed top-up flows.

This module covers:

- execution billing lifecycle (`reserve` / `consume` / `refund`)
- user top-up lifecycle (`topup`, webhook completion, history)
- payment provider capability discovery (`/payments/config`)
- deployment modes (paid, free, unlimited)

## Core Data Contracts

### User balance state

- `credits_balance`: immediately spendable credits
- `reserved_credits`: credits locked for in-flight executions
- `total_credits_used`: cumulative consumed credits

### Ledger state (`credits_transactions`)

Each mutation writes an immutable row with:

- transaction type
- deltas (`amount`, `reserved_delta`, `total_used_delta`)
- optional `execution_id`
- metadata (provider/session/order identifiers, pricing, debug context)

## Interaction Flows

### Execution billing

1. Plane reserves estimated credits before dispatch.
2. Runtime reports completion and actual usage.
3. Plane settles using one atomic DB transaction:
   - success: consume actual usage and release reserve
   - failure/cancel: release reserve and refund unused amount

### User top-up

1. Frontend loads provider capabilities from `GET /payments/config`.
2. User selects amount and provider, then calls `POST /credits/topup`.
3. Paid providers (Stripe/PayPal) return a checkout URL and stay `pending` until webhook completion.
4. Free/manual paths can complete immediately and return updated balance directly.
5. Webhook completion updates payment status and applies credits atomically.

## Provider Coverage

- Stripe card checkout
- Stripe Alipay checkout
- Stripe WeChat Pay checkout (`wechat_pay.client=web`)
- PayPal checkout
- Optional mock provider for local/dev environments
- Manual top-up behavior when payments are disabled

Provider exposure is runtime-configurable and reflected by `/payments/config`.

## Deployment Modes

Credits, payment availability, wallet methods, tier multipliers, starter bonus,
and billing rates are operational policy. Manage them in Admin -> System ->
Platform policy or through `GET/PUT /admin/system/policy`.

- Paid mode: `credits_enabled=true`, `payment_enabled=true`, enable only the
  intended provider methods, and configure provider credentials as deployment
  secrets.
- Credits-only mode: `credits_enabled=true`, `payment_enabled=false`; users run
  against balances managed by starter bonus, admin grants, and refunds.
- Unlimited mode: `credits_enabled=false`; execution bypasses reserve/consume
  enforcement.

## API Surface

- `GET /credits/balance`
- `GET /credits/history`
- `POST /credits/topup`
- `POST /credits/estimate`
- `POST /credits/reserve` (service HMAC only)
- `POST /credits/consume` (service HMAC only)
- `POST /credits/refund` (service HMAC only)
- `POST /credits/webhook/stripe`
- `POST /credits/webhook/paypal`

## Production Checklist

- enforce webhook signature verification in all non-local environments
- keep payment completion and credit grant in a single transaction
- enable alerting on repeated webhook failures
- expose only intended providers through platform policy
- keep user credit grants limited to provider webhooks or administrator audit endpoints

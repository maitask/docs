# Implementation Audit

## Scope

This page defines how implementation completeness is reviewed across:

- `plane-frontend`
- `plane`
- `runtime`

It is a living release gate, not a historical validation snapshot. A capability
is considered complete only when code, storage, API contracts, UI behavior,
tests, documentation, and deployment behavior all match.

## Evidence Requirements

Each audited capability must have:

- source implementation in the owning component
- typed request/response or package input/output contract
- persistence or runtime state model when data is stored
- authentication, authorization, ownership, and service-auth checks where needed
- operator-visible state for operational features
- test or smoke coverage proportional to risk
- documentation that describes implemented behavior only

## Component Coverage

### Plane Frontend

Plane Frontend owns public pages, app shell, ops shell, workflow UX, package
marketplace, execution views, scheduling, credits, billing, notifications,
settings, and admin/runtime management screens.

Release criteria:

- UI actions map to real Plane APIs
- all user-facing text is internationalized
- loading, empty, error, and permission states are implemented
- admin and user surfaces show only data relevant to that audience
- runtime and policy management screens reflect persisted backend state

### Plane

Plane owns auth, users, packages, executions, credits, payments, schedules,
workflows, notifications, runtime control, analytics, audit, and admin APIs.

Release criteria:

- APIs use the unified response/error contract
- credits and payments are transactional and auditable
- operational policy is stored in system settings and managed through admin API/UI
- service-auth and ownership checks protect internal mutation routes
- privileged, billing, package, runtime, and security changes are audit logged

### Runtime

Runtime owns package execution, worker polling, sandbox policy, adapters,
release artifacts, update installation, and host-local execution state.

Release criteria:

- package execution uses explicit input/output contracts
- worker polling and callbacks tolerate retries, restarts, and network loss
- workspace, network, adapter, and resource policies are enforced in runtime code
- native release artifacts are verified before installation
- update status is reported to Plane with actionable failure details

## Cross-Service Chain

The formal execution chain is:

1. Plane Frontend calls Plane APIs.
2. Plane authenticates, authorizes, applies platform policy, and persists state.
3. Plane dispatches execution through the Runtime gateway or runtime worker queue.
4. Runtime executes packages and reports status/results.
5. Plane settles credits, stores outputs, emits notifications, and exposes state
   to the frontend.

## Release Verdict

Do not use this page to declare a permanent go-live status. The release verdict
belongs to the target-environment checklist in `operations/release-readiness`.

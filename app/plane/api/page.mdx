# Plane API Contract

## Base URL

- Development: `http://localhost:8001`
- Production: deployment-specific API domain

## Authentication Modes

- **User auth**: bearer token from `/auth/login` or `/auth/register`
- **API key auth**: token endpoints under `/auth/tokens`
- **Service auth**: HMAC headers for trusted service calls (Runtime to Plane)

## Core Endpoint Domains

1. `auth`: register, login, refresh, logout, verification, API keys
2. `users`: profile, account updates, installed package links
3. `packages`: browse/search/get/publish/update/delete/share/visibility
4. `executions`: create/get/list/update/cancel and runtime result syncing
5. `workflows` and `schedules`: lifecycle and manual execution
6. `credits` and `payments`: balance, reserve/consume/refund, top-up
7. `notifications`: status, list, create, preferences

## Credits And Payments Contract

### Capability discovery

Frontend must load provider capabilities from:

- `GET /payments/config`

Response fields used by UI:

- `supported_card_brands`
- `supported_providers[]` (`stripe`, `paypal`, optional `alipay`, `wechat_pay`)
- `payment_enabled`
- `mock_enabled`

### Top-up behavior

- Endpoint: `POST /credits/topup`
- `payment_method` accepts:
  - provider id (`stripe`, `paypal`, `alipay`, `wechat_pay`, `mock`, `manual`)
  - saved payment method UUID
- Response behavior:
  - `payment_url` exists: external checkout flow (Stripe/PayPal)
  - `payment_url` absent: direct credit top-up completion (free/manual/mock modes)

### Webhook completion

- `POST /credits/webhook/stripe`
- `POST /credits/webhook/paypal`

Webhook processing requirements:

- signature verification
- event persistence (`payment_webhooks`)
- idempotent payment settlement
- atomic credit grant and balance mutation

### Service-authenticated credit mutations

These endpoints support Runtime->Plane service authentication:

- `POST /credits/reserve`
- `POST /credits/consume`
- `POST /credits/refund`

## Operational Endpoints

- `GET /health`, `/health/ready`, `/health/live`
- `GET /metrics`
- `GET /api-docs/openapi.json`
- `GET /swagger-ui`

## Response Shape

Plane uses a unified envelope:

```json
{
  "success": true,
  "data": {},
  "pagination": {
    "page": 1,
    "per_page": 20,
    "total": 100,
    "total_pages": 5
  }
}
```

## Release Notes For Integrators

- Treat `/payments/config` as source of truth; do not hardcode providers.
- In free mode, absence of `payment_url` from `/credits/topup` is a success path, not an error.
- Keep admin-only and user-scoped routes protected by backend auth checks; frontend visibility checks are not sufficient.
